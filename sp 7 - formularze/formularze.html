Formularze

1

Struktura i przesyłanie danych

  • form zbiera pola i wysyła je na serwer.
  • action wskazuje adres endpointu.
  • method określa sposób przesyłania: GET lub POST.
  • fieldset i legend porządkują pola semantycznie i wizualnie.
  • Dane wrażliwe (hasła, dane osobowe) wysyłamy przez POST, nigdy przez URL.
<form action="/konto/rejestracja" method="post">
  <fieldset>
    <legend>Dane konta</legend>
    ...
  </fieldset>
</form>
Przykład GET - wyszukiwanie
Przykład POST - dane wrażliwe
2

Pola wprowadzania danych - typy input

  • Podstawy: text, password, email, number.
  • Wybory: checkbox i radio, grupowane wspólnym atrybutem name.
  • HTML5: date, color, range, tel, search, url – lepszy UX i klawiatury mobilne.
  • Listy rozwijane: select z option; atrybut multiple pozwala wybrać kilka opcji naraz.
  • hidden – niewidoczne pole przesyłające dane (np. token CSRF); nie renderuje się w interfejsie.
  • textarea – wieloliniowy tekst; rows określa wysokość, wartość domyślna to treść elementu (nie value), resize kontrolowane przez CSS.
<input type="email" name="kontakt_email" />
<input type="radio" name="plan" value="pro" />
<select name="kraj">...</select>
<textarea name="wiadomosc" rows="4" maxlength="500"></textarea>
Checkbox
Radio (ten sam name = jedna grupa)
Hidden – niewidoczne, nie renderuje się w interfejsie

<input type="hidden" name="csrf_token" value="abc123" />

3

Dostępność i UX

  • Label łączymy z polem przez for i id.
  • Kliknięcie etykiety aktywuje pole i poprawia wygodę.
  • Placeholder to podpowiedź, ale nie zastępuje labela.
  • Pomocne atrybuty: readonly, disabled, autofocus.
<label for="nick">Nick</label>
<input id="nick" name="nick" autofocus />
<input value="nieedytowalne" readonly />

Placeholder znika po wpisaniu tekstu i może być słabo widoczny. Etykieta powinna zostać stale widoczna.

4

Walidacja po stronie klienta

  • Wbudowane atrybuty: required, minlength, maxlength, min i max.
  • pattern pozwala wymusić prosty format regex.
  • Przeglądarka sama pokazuje komunikaty błędów bez JavaScriptu.
  • novalidate na formularzu wyłącza wbudowaną walidację przeglądarki (np. do testów lub własnej walidacji JS).
<input required minlength="3" maxlength="20" />
<input pattern="[A-Za-z]{2}[0-9]{4}" />
<form novalidate>...</form>
5

Przyciski i interakcja z formularzem

  • button type="submit" wysyła formularz.
  • button type="reset" czyści pola do stanu początkowego.
  • button type="button" uruchamia akcje pomocnicze (np. JS), ale nie wysyła formularza.
  • Przesyłanie plików wymaga enctype="multipart/form-data".
<form method="post" enctype="multipart/form-data">
  <input type="file" name="avatar" />
  <button type="submit">Wyślij</button>
</form>
6

Dodatki

  • datalist podpowiada wartości bez ograniczania wpisu.
  • output działa jako semantyczny pojemnik na dane
  • meter i progress wizualizują odpowiednio stan i postęp - różnica semantyczna
  • autocomplete i inputmode poprawiają UX, zwłaszcza na mobile.
<input list="miasta" name="miasto" />
<datalist id="miasta">...</datalist>
<meter min="0" max="100" value="72">72%</meter>

Stylowanie formularzy CSS

  • :focus-within na .field podświetla etykietę gdy dowolne pole w środku ma focus.
  • :read-only wizualnie odróżnia pola tylko do odczytu od edytowalnych i disabled.
  • ::file-selector-button pozwala ostylować przycisk wyboru pliku jak każdy inny element.
  • appearance: none usuwa natywny wygląd checkbox i radio — pełna kontrola z CSS.
  • Pseudo-klasy :focus, :valid i :invalid poprawiają informację zwrotną dla użytkownika.
.field:focus-within label { color: blue; }
input:read-only { background: #f0f4f8; }
::file-selector-button { border-radius: 999px; }
input[type='checkbox'] { appearance: none; }
input[type='checkbox']:checked { background: var(--accent); }
input:valid { border-color: green; }
input:invalid { border-color: red; }
